The Concept

Not a dating app. Not a social feed. Something new.

HereNow is a location-aware social layer for the physical world. When you walk into a bar, a coffee shop, or an event — you check in, show your Social Mode (Dating / Friends / Networking / Just Vibes) and your Mood (Open / Selective / Not Today). Everyone at that venue sees who's there and why. No endless swiping. No algorithmic feed. Just real people, in real places, right now.

Core System

How it works

Social Mode

Why you're out

Set before you check in. Shown to everyone at the venue. One of: Dating, Friends, Networking, or Just Vibes. Removes ambiguity. You know exactly who's open to what.

Dating Friends Networking Just Vibes
Mood Mode

How approachable you are

Separate from Social Mode. Open = come say hi. Selective = be thoughtful about it. Not Today = hard boundary, don't approach. Shown as a small indicator on every person card.

Open Selective Not Today
We Met

The IRL handshake

After meeting someone in person, both parties confirm "We Met." That mutual confirmation unlocks direct messaging. No We Met = no DMs. It keeps it real. Expires 4 hours after your session ends.

Afterglow

The session recap

When you check out, you get an auto-generated recap: how long you were there, how many people were in the venue, how many We Mets you made. The reflection moment after a real night out.

Roadmap

What comes next

PHASE 1 — Web First
Web

Onboarding Flow

First-run experience explaining Social Mode, Mood Mode, and We Met before the first check-in. Single most important UX investment — users who don't understand the system won't engage. Must be done before any outside user touches the app.

Web

Profile Photos

Image picker + Supabase Storage upload for avatars. The People list can't feel real with placeholder initials. This is table stakes for any social app and needs to be live before real users try it.

Web

Map View

Replace the text zone list with a real interactive map. Venue pins with heat color per activity level. Makes the core discovery experience feel like an actual product instead of a list.

Web

Safety Layer

Report / block system. Fortress Mode (invisible — you see others but no one sees you). Non-negotiable before any user outside the team is invited. Location-based social apps need this before anything else goes public.

Web

Push Notifications

We Met requests, new DMs, event reminders. Without push, there is no re-engagement loop. People check in and never come back because nothing pulls them back. This closes that gap.

Web

Premium DMs + Stripe

Free DMs expire at 72 hours. Pay to extend to 14 days. The paywall hits at the exact right moment — after a real connection is already made. Stripe integration required. This is revenue #1.

PHASE 2 — Launch
Launch

Nashville Soft Launch

Seed 3–5 specific venues (Lower Broadway, 12South, East Nashville bar scene). Get 20–30 real users checking in over a weekend. See what breaks and what clicks. This is the first real validation.

Launch

Venue Accounts

Business owners claim their zone, verify it, post official events, see crowd analytics. Revenue #2. Bars and coffee shops benefit directly from knowing their real-time crowd — that's a natural sell.

Launch

Deep Links + Sharing

Share a zone link that opens the app. Share Afterglow recap as an image. Required for organic growth — people need to be able to say “come join me at [venue]” and have it just work.

Launch

Analytics

Check-ins per day, We Met conversion rate, DM expiry hit rate, venue heat trends. Instrument before launch so every decision is data-backed, not guesswork.

PHASE 3 — Native (After Web is Proven)
Native

EAS Build (Android + iOS)

Once the web version is fully functional and tested, use Expo Application Services to generate signed APK/AAB for Android and IPA for iOS. One codebase, two stores. This comes after the product is proven, not before.

Native

Google Play + Apple App Store

Submit to Google Play internal testing ($25 one-time developer fee) and Apple TestFlight ($99/yr developer account). Full public store listings come after the closed test phase passes. This is the last milestone, not the first.

Session bug sweep — June 2026

12 issues fixed in a single focused session. Listed for accountability and future reference:

✅ kickoff_prompt column mismatch — Profile edit and profile tab both referenced a non-existent column, causing the entire profile to load blank. Fixed to use the correct kickoffs text[] schema column across all three files.
✅ DM window always expired — confirmWeMet() was not updating expires_at, leaving it set to the original 4-hour request window. Fixed to reset to 72 hours from confirmation time.
✅ Not Today bypass possible — UI hid the We Met button but the lib made no server-side check. sendWeMet() now fetches recipient profile and returns null before inserting if mood_mode is 'not_today'.
✅ Afterglow highlights always empty — checkOut() never populated the highlights text[] column. Now generates human-readable highlights (connections made, people in space, time duration).
✅ New user dead end — Signup routed directly to the tab bar, leaving users with blank PersonCards everywhere. Now routes through /profile/edit first.
✅ Profile edit back button stranded users — Back button and save both used router.back(), which would throw users back to the auth stack if they arrived from signup. Fixed to use router.replace('/(tabs)').
✅ 'use client' in React Native — login.tsx had a stale Next.js directive that is meaningless and potentially harmful in Expo. Removed.
✅ No safety features — No report, no block, no ⋯ menu anywhere in the app. Built from scratch: lib/reports.ts, lib/blocks.ts, ⋯ menu on PersonCard, block filtering in zone view, safety_reports + user_blocks tables in Supabase.
✅ Post likes not tracked — Like button had no state, no optimistic update, and no persistence. Built lib/posts.ts with toggleLike() and fetchLikedPostIds(). Feed now hydrates liked status in a single batch query.
✅ Afterglow screen had no highlights UI — The highlights array was being generated but never displayed. Added a highlighted moments section with cyan bullet points to the Afterglow screen.
✅ Zone view showed blocked users — After building the block system, the zone view needed to actually filter them. loadPeople() now fetches blocked IDs in parallel and excludes them from the rendered list.
✅ Afterglow Afterglow display missing session context — Interface type updated to include the highlights field so the screen can receive and render what was generated on checkout.

Infrastructure & Costs

What it actually costs to run HereNow at scale

HereNow is not a simple CRUD app. It runs three independent infrastructure layers simultaneously, powers real-time venue session streams, processes anti-cheat signals per check-in, serves live analytics dashboards to venue owners, and handles ephemeral data pipelines across every active city. This is a full operational platform. The cost model reflects that — and so does the revenue model on the other side.

The Three Infrastructure Layers

Each layer scales independently. Each has its own failure boundary. None is a bottleneck to the others.

LAYER 1 — MEMBER APP

iOS, Android, and Web PWA. Lightweight state-based client. Downloads minimal data. Subscribes to real-time venue session streams. Handles check-in UI, Social/Mood Mode, People List, Pulse, Heat Map, We Met, DMs, and Afterglow.

LAYER 2 — VENUE PORTAL

Separate web dashboard for venue owners and managers. Real-time crowd analytics, Crowd Density Curves, Social/Mood Mode breakdowns, We Met counts, Pulse moderation, Event management, Promotion tools, Staff logins, and Tier management. Runs independently of the member app.

LAYER 3 — CORE ECOSYSTEM

The back-end brain. Handles: presence verification pipeline, venue session lifecycle, real-time Heat Map aggregation, analytics time-series storage, anti-cheat confidence scoring, AI moderation layer, DM expiry jobs, Afterglow generation, push delivery, and city-cluster coordination.

What Drives Cost as HereNow Grows

SystemWhy It Costs MoneyCost Behavior
Real-Time Venue Sessions Each active venue runs its own isolated session stream. Every checked-in user subscribes to ~4–5 real-time channels simultaneously (People List, Chat, Pulse, We Met, DMs). This is the single biggest cost driver. Scales with simultaneous active venues & users
Venue Portal Analytics Crowd Density Curves, Social/Mood Mode breakdowns, We Met counts, Pulse sentiment, dwell time tracking, and Heat Map prediction require time-series data storage and live aggregation pipelines per venue. Scales with number of active venue subscriptions
Anti-Cheat Processing Every check-in runs GPS geofencing, WiFi signature matching, BLE beacon correlation, motion pattern scoring, and device integrity checks. Produces a presence confidence score per user per session. Per check-in, not per minute — manageable
AI Moderation Live Venue Chat, Pulse posts, and DMs must be scanned in real time. Language moderation, tagging interception, and Safety Score adjustments all run continuously during active sessions. Scales with active sessions & message volume
Heat Map Aggregation City-level Heat Map requires constantly updating intensity values across all active venues, smoothed at 30–60 second intervals. Computationally heavier than a simple location ping. Throttled intervals keep cost predictable
Storage (Photos) Profile photos and venue covers accumulate permanently. Pulse media is ephemeral (12hr auto-delete). Live Chat deletes on session end. Only profile and venue photos compound over time. Slow, predictable growth — manageable
City-Based Clusters Each city runs as a semi-independent shard: its own venue DB, Heat Map, event ecosystem, moderation pool, and analytics models. Cross-city overhead is minimal (identity, auth, subscriptions only). Step-function cost increase per new city launch
Push Notifications We Met alerts, DM pings, venue event announcements, session changes. Delivered via Expo Push infrastructure. Free — no meaningful cost at any scale
Maps API Venue map with pins, Heat Map overlay, geofencing polygons for each venue's exact footprint. Free up to ~28k loads/month. Can switch to OpenStreetMap (free forever) with minimal code change. Free until meaningful city scale

Venue Portal — B2B Dashboard Infrastructure

The Venue Portal is a separate web application that runs independently from the member app. Venue owners log in to see live occupancy, analytics, and management tools. It has its own infrastructure costs on top of the member app backend.

ComponentWhat It DoesCost Driver
Live Occupancy Stream Venue dashboard shows a real-time count of checked-in members, Social/Mood Mode breakdown, and People List updates as users arrive and leave. Each open dashboard tab holds a persistent Realtime subscription. Scales with number of venue owners logged in simultaneously
Crowd Density Analytics Historical occupancy curves, busiest hours heatmap, average dwell time, peak headcount by day/week. Bottom line on Venue Portal cost: At launch with 1–10 venues, the portal adds essentially nothing to infrastructure spend — it runs on the same Supabase database and Realtime connections as the member app. At 100+ active venues with dashboards open simultaneously, Venue Portal becomes the second-largest cost driver after member real-time sessions. The Engage ($99/mo) and Amplify ($249/mo) subscription tiers are priced to cover this overhead with strong margin.
